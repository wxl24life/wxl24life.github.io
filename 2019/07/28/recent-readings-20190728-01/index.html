近期浏览器阅读列表 (07-28)

发表于 2019-07-28 | 阅读次数:

题记：电脑若干天没关机，浏览器又.. 打开了一堆页面，大多数页面都「舍不得」关掉。所以决定在关闭之前把读过觉得有收获的几个网页做个简单摘要，同时也是作为备忘记录一下。

回想起一个多月前发表在「码农翻身」知识星球上的一段话。

早上在微信主动搜索到的一篇关于注意力的文章。结合文章联想下我自己的「注意力漂移」坏习惯。经常在搜索某主题的时候，不断打开新的网页，然而又「舍不得关掉」，而且多数时候容易「跑题」，比如某网页里面提到了某个与当前搜索主题无关的内容，因为「好奇心」，马上去点开链接或者去搜索框搜索，以至于打开的网页越来越多。

在想有没有这样的浏览器插件，如果没有做一个应该也不难。它是一个浏览器报警插件，功能很简单：「当同时打开的网页数量超过 xx 个的时候请给我报个警」。xx 默认设置成 10 个。

附上链接：李笑来：注意力飘逸

最近在思索和探究一个问题：一个工业级的开源分布式系统，从开发到具备系统级运行之前是如果做测试的？带着这个疑问，我把注意力重新放回到了 kafka 的测试上面。

  1. How Apache Kafka is Tested

    Confluent 官方 blog 上面的文章质量都很高，如果对 kafka 感兴趣，建议经常翻翻上面的文章。这篇文章发表于 2017 年 9 月。文章首先提出对 kafka 做测试的必要性，尤其是 correctness & performance，同时，也客观地指出在分布式系统环境下追踪排错的困难。之后详细介绍了 Confluent 和 kafka 社区在保证 kafka 正确定和性能方面所做的工作。包括从每个特性最初的设计开始，重大的 feature 首先需要有详细的 KIP 提议文档，经由社区进行审核讨论。

    之后的代码审核是非常重要的一环，kafka 社区以实践证明，花在代码审核上面的时间是值得的，此处作者引用了一篇论文（稍后做介绍），这篇论文指出代码中对错误异常处理不当往往是引起严重系统错误的主要原因。

    最后从单元测试、集成测试、分布式系统级测试以及社区用户提交的 bug 等四个维度介绍了 kafka 是如何进行测试的。系统级测试方面提到了一个框架 ducktapes，它可以模拟分布式环境下各种复杂异常状况。同时，Confluent 每天都会对 kafka 在不同系统测试场景下进行大量测试，测试结果 都是公开的。

  2. Simple Testing Can Prevent Most Critical Failures

    这篇论文的链接引用自本文第一篇文章。论文整理了几大开源分布式系统的 Bug List，通过对 bug 产生原因的分析，总结出严重的系统级 bug 往往来自于代码中对错误异常的不恰当处理。由于论文中指出了 HBase 里存在的若干 bug，论文发表后，HBase 社区专门针对论文提到的问题进行了集中 整治。这篇论文被很多文章引用，第一作者是多伦多大学的年轻学者 DingYuan，本科毕业于北航。论文原文没有读，看完了他在 OSDI14 会议上的 演讲 和 slides，英语做的 talk 非常赞。

  3. Trogdor: Kafka Fault Injection

    kafka 社区提供的压测工具。

  4. KIP-480: Sticky Partitioner

    kafka 社区近期的一个 KIP，里面讲到了性能优化，重点是对文档里面测试环节的关注，对 kafka 系统做性能测试的最初想法也来源于此。这月初让项目组入职新人把测试部分在我们自己的集群环境下做了复现，主要目的是熟悉测试工具的使用。
